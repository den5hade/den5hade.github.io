Многие считают, что для построения Event-Driven Architecture (EDA) обязательно нужен Apache Kafka или RabbitMQ. Но если ваша нагрузка измеряется тысячами, а не миллионами событий в секунду, PostgreSQL уже содержит всё необходимое.

В этой статье мы разберем, как использовать **Логическую репликацию** для реализации паттерна **Transactional Outbox**.

## Почему не Kafka?

Kafka — это мощно, но дорого в обслуживании. Главная проблема классических брокеров — **атомарность**. Вам нужно одновременно обновить базу и отправить сообщение в очередь. Если база обновилась, а брокер упал — данные разошлись.

В Postgres мы пишем событие в ту же транзакцию, что и бизнес-данные. А затем «вычитываем» его из логов репликации.

## 1. Настройка базы данных

Сначала переведем Postgres в режим генерации логических логов. В `postgresql.conf` установите:

```conf
wal_level = logical
max_replication_slots = 5
```

Создадим таблицу для событий (Outbox):

```sql
CREATE TABLE event_outbox (
    id UUID PRIMARY KEY,
    event_type TEXT NOT NULL,
    payload JSONB NOT NULL,
    created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);

-- Создаем слот репликации
SELECT * FROM pg_create_logical_replication_slot('outbox_slot', 'pgoutput');
```

## 2. Генерация событий в FastAPI

В вашем приложении на Python/FastAPI мы просто сохраняем событие в ту же транзакцию:

```python
from fastapi import FastAPI, Depends
from sqlalchemy.orm import Session

@app.post("/orders")
def create_order(order_data: OrderSchema, db: Session = Depends(get_db)):
    # 1. Создаем заказ
    new_order = Order(**order_data.dict())
    db.add(new_order)
    
    # 2. Пишем событие в ту же транзакцию
    event = EventOutbox(
        event_type="order_created",
        payload={"order_id": new_order.id, "total": new_order.total}
    )
    db.add(event)
    
    db.commit() # Событие попадет в WAL только при успешном коммите
    return {"status": "ok"}
```

## 2. Python Worker: Читаем WAL

Для чтения логов используем библиотеку psycopg. Нам нужен отдельный процесс-воркер, который будет «слушать» поток изменений:

```python
import psycopg
from psycopg.rows import dict_row

def run_worker():
    conn_str = "dbname=mydb user=postgres password=pass host=localhost"
    
    # Подключаемся в режиме репликации
    with psycopg.connect(conn_str, replication="database") as conn:
        cursor = conn.cursor()
        
        # Начинаем стриминг из слота
        cursor.start_replication(
            slot_name="outbox_slot",
            plugin_name="pgoutput",
            options={"proto_version": "1", "publication_names": "all_tables"}
        )

        print("Worker started, waiting for events...")

        while True:
            msg = conn.receive_message()
            if msg:
                # Здесь происходит магия декодирования данных
                # После обработки отправляем подтверждение (LSN)
                conn.send_replication_feedback()
```

## Преимущества подхода

1. Guaranteed Delivery: Событие никогда не потеряется, если транзакция прошла успешно.
2. No Data Drift: Состояние базы и очередь всегда синхронизированы.
3. Infrastructure Simplicity: Вам не нужно мониторить кластер Kafka и Zookeeper. Только одна база данных.

> Postgres — это отличный «брокер сообщений на минималках», который закрывает потребности 90% проектов. Логическая репликация дает нам гарантии доставки уровня Enterprise без усложнения стека.
