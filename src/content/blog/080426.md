Механизм **LISTEN / NOTIFY** в PostgreSQL — это встроенная система событийно-ориентированного взаимодействия (Pub/Sub). Она позволяет базе данных «крикнуть» в определенный канал, а вашему Python-приложению — мгновенно услышать это и выполнить код (например, отправить письмо).

## Как это работает (алгоритм)

1. **В базе данных**: Вы создаете триггер на таблицу `users`. Как только происходит `INSERT`, триггер выполняет команду `NOTIFY`, передавая в качестве сообщения (payload) ID нового пользователя.
2. **В Python-приложении**: Запускается отдельный асинхронный процесс (воркер), который открывает соединение с БД и выполняет `LISTEN`.
3. **Связь**: Когда воркер получает уведомление, он вычитывает данные пользователя и отправляет Email.

---

## Плюсы и минусы

| Преимущество | Описание |
|-------------|----------|
| ✅ **Скорость** | Это почти мгновенно (real-time). Не нужно постоянно опрашивать базу запросами `SELECT * FROM...` |
| ✅ **Простота** | Не нужны Redis, RabbitMQ или Kafka. Все живет внутри Postgres |

| Недостаток | Описание |
|-----------|----------|
| ❌ **Риск потери** | Если в момент `NOTIFY` ваш Python-воркер был выключен (перезагрузка, сбой), сообщение пропадет. `NOTIFY` не хранит историю |
| ⚠️ **Решение проблемы потерь** | Обычно этот метод комбинируют с проверкой таблицы при старте воркера: найти всех `is_active: false`, кому еще не ушло письмо |

## Пример реализации (на базе asyncpg)

### 1. Подготовка в базе данных (SQL)

Создаем функцию и триггер, который будет срабатывать при регистрации:

```sql
-- Функция, которая отправляет уведомление
CREATE OR REPLACE FUNCTION notify_user_created()
RETURNS TRIGGER AS $$
BEGIN
  -- Канал 'new_user_event', полезная нагрузка - ID пользователя
  PERFORM pg_notify('new_user_event', CAST(NEW.id AS TEXT));
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

-- Привязываем функцию к таблице users
CREATE TRIGGER user_after_insert
AFTER INSERT ON users
FOR EACH ROW
EXECUTE FUNCTION notify_user_created();
```

### 2. Воркер на Python (через asyncpg)

Библиотека asyncpg идеально подходит для прослушивания каналов:

```python
import asyncio
import asyncpg

async def send_email(user_id):
    # Логика отправки письма через SMTP или API
    print(f"Отправка письма пользователю с ID: {user_id}")
    await asyncio.sleep(1) 

async def worker():
    # Подключаемся к БД
    conn = await asyncpg.connect(dsn="postgresql://user:pass@localhost/db")

    # Функция-обработчик события
    def callback(connection, pid, channel, payload):
        user_id = payload
        print(f"Получено уведомление! ID пользователя: {user_id}")
        # Запускаем отправку письма в фоне, чтобы не блокировать прослушивание
        asyncio.create_task(send_email(user_id))

    # Начинаем слушать канал
    await conn.add_listener('new_user_event', callback)
    
    print("Воркер запущен и слушает канал 'new_user_event'...")
    
    # Держим соединение открытым бесконечно
    try:
        while True:
            await asyncio.sleep(1)
    finally:
        await conn.close()

if __name__ == "__main__":
    asyncio.run(worker())
```

Чтобы гарантировать обработку всех событий даже после перезагрузки воркера, добавьте в таблицу `users` флаг-индикатор (например, `is_email_sent BOOLEAN DEFAULT FALSE`) и временную метку создания записи. При старте воркер должен выполнять «дозаказ»: выбирать все записи, где `is_email_sent = FALSE` и `created_at` старше допустимого окна, и обрабатывать их в фоновом режиме. Такой подход компенсирует потерю уведомлений `NOTIFY` во время простоя, обеспечивая надёжность доставки без усложнения архитектуры.
